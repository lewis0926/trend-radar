Trend Radar is a real-time dashboard identifying which industries are trending based on past price action.

## Project
Track a list of ETFs/indices with exposure to all major market industries — tech, healthcare, finance, energy, consumer discretionary, and more.

## Tech Stack
- **Backend**: Python 3.10+, FastAPI + uvicorn, venv for dependency management
- **Frontend**: Vite + React + TypeScript (`frontend/`), Bun for package management
- **Config**: Pkl — generate `config.json` from a Pkl file and parse it in the app

## Config Structure
- `config/schema/Config.pkl` — typed schema with defaults. All fields are defined here.
- `config/<env>.pkl` — environment-specific file (e.g. `dev.pkl`). Amends the schema and supplies env-specific values like API keys.
- `config.json` — generated by running `pkl eval "config/<env>.pkl" -f json -o "config.json"`. This is what the app reads.

## Running
```bash
# Backend API (from backend/)
cd backend && source ../venv/bin/activate
uvicorn src.api:app --reload

# Frontend (from frontend/)
cd frontend && bun run dev
```

## Adding a New ETF
1. Add the typed entry to `backend/config/schema/Config.pkl`
2. Override or extend in `backend/config/<env>.pkl` if needed
3. Regenerate: `cd backend && pkl eval "config/dev.pkl" -f json -o "config.json"`

## Deployment
Build the Docker image and deploy as a container. The frontend is a static build served separately.

## Coding Standards
- `config.json` is the single config file. All configuration — including secrets like API keys — goes in env Pkl files and flows into `config.json` (which is gitignored). No `.env` files, no environment variables.
- All Python functions must have explicit type annotations on parameters, return values, and newly defined variables.

## Frontend Coding Standards

**Component structure**
- Each logical UI section gets its own component file in `frontend/src/components/`.
- Components with non-trivial styles get a co-located `.css` file (e.g. `Header.tsx` + `Header.css`).
- `App.tsx` is a thin orchestrator: data fetching + layout only, no styling.

**CSS over inline styles**
- All styling goes in CSS files — no inline `style={{}}` props except for values that are purely data-driven at runtime (e.g. a bar width from a percentage, a border color from a positive/negative value).
- Design tokens (colors, fonts) live as CSS custom properties in `index.css` under `:root`. Reference them as `var(--orange)` etc. — never hardcode hex values in component CSS or JS.
- Common utility classes (`.mono`, `.tabular`, `.positive`, `.negative`) are defined in `index.css` and composed in JSX `className`.
- Shared structural patterns (`.section-title`, `.page`) are defined in `index.css`; component-specific styles go in the component's own CSS file.
